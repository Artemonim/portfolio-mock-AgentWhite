# **Проект AgentWhite — среда bug bounty исследований под управлением ИИ-команды**

[In English](./README_ENG.md)

## **Краткий обзор**

`AgentWhite` — монорепозиторий операционной среды для авторизованных bug bounty исследований, где ИИ работает не как чат-ассистент, а как управляемая команда специализированных агентов. Центральный технический элемент — Rust-библиотека и CLI (`agentwhite-core` / `agentwhite-cli`), реализующие **policy gateway**: принудительное соблюдение правил программы на сетевом уровне. Журнал исследования (`research-log.md`) — постоянная хронологическая память всей кампании: единый источник истины о том, что проверено, что исключено и какие гипотезы активны.

*Код проекта является закрытым. Конкретные руководства оператора (runbooks), маршрутизация промптов, внутренние эвристики и специфичные для программ рабочие процессы не раскрываются.*

*Github Langbar симулирован при помощи [gh-lang-mock](https://github.com/Artemonim/gh-lang-mock) по распределению языков исходного репозитория.*

## **Технологический стек**

*   **Rust core**: workspace `crates/*`, `resolver = "3"`, edition 2024, Rust 1.85+, `unsafe_code = "forbid"`; `agentwhite-core` (policy engine, HTTP proxy, аудит) + `agentwhite-cli` (CLI на `clap`: `preflight`, `request`, `proxy`, `tool`)
*   **TypeScript tooling**: `pnpm` workspace, Node `>=24.14`, ESLint + `typescript-eslint`; строгий ESM, ES2024, `verbatimModuleSyntax`; типизированные веб- и инструментальные интеграции
*   **Python subsystem**: Python 3.13+, Ruff, MyPy (strict), pytest; аналитика и вспомогательная автоматизация
*   **PowerShell orchestration**: `run.ps1` → `tools/ci/build.ps1`; оркестратор стадий CI, recon smoke и JSON-отчёт
*   **Go utilities**: модуль `agentwhite.local/go`; точечные утилиты
*   **Внешний security tooling** (`tools/third_party/`):
    *   recon / OSINT: `subfinder`, `gau`, `asnmap`
    *   target-side / HTTP / fuzzing: `httpx`, `katana`, `ffuf`, `nuclei`, `naabu`
    *   orchestration-friendly runtime: `BBOT`, Docker-сценарии; субмодуль `nuclei-templates`
*   **Cursor Browser MCP**: пассивное наблюдение за поверхностью — агент видит страницы как пользователь и собирает навигационные артефакты без отправки security-специфичных запросов
*   **Операторские правила**: `AGENTS.md`, `Techlead.mdc`, `SubagentRules.md` — основаны на [AgentCompass](https://github.com/Artemonim/AgentCompass)
*   **CI/CD**: локальный PowerShell-оркестратор `run.ps1`, стадийные скрипты в `tools/ci/`, машинно-читаемый JSON-отчёт

## **Ключевые задачи и решения**

*   **Принудительное соблюдение правил программы при автоматизации:** Автоматизированное исследование может нарушить границы области, лимиты запросов или раскрыть идентичность оператора перед целью. Решение — Rust policy gateway: каждый запрос проходит через per-программный `policy.toml` (scope CIDR/домены, rate limit, обязательные заголовки, VPN-маркеры), инструменты с поддержкой proxy маршрутизируются через него принудительно. Несоответствие policy — отказ до отправки пакета.
*   **Двухрежимный доступ к исследовательской поверхности:** Не каждый исследовательский шаг требует контролируемого запроса. Решение — разделение на два режима: **Cursor Browser MCP** для наблюдения за страницами (агент видит то, что видит пользователь, собирает навигационные артефакты без дополнительного трафика) и **agentwhite gateway** для специфических security-запросов, проходящих через policy enforcement. Браузер — пассивные глаза команды; gateway — управляемые руки.
*   **Накопление исследовательской памяти между сессиями:** Агенты стартуют с нулевой памятью, а кампания может длиться несколько недель. Решение — `research-log.md` как хронологическая постоянная память: каждый цикл пишет туда проверенные гипотезы, убитые векторы, активные priors и ключевые решения. Новые субагенты получают выдержки из лога в промпт; насыщенность лога — сигнал к проектированию следующего `TODO.md`.
*   **Дисциплина артефактов при работе с чувствительными данными:** Сырые результаты сканирования и промежуточные находки не должны попасть в git или публичный слой. Решение — жёсткое разделение: `.temporary/<program>/` (gitignored, эфемерное хранилище), `evidence/` (очищенные артефакты, git-safe), `reports/` (только шаблоны раскрытия), `.enforcer/` (аудит gateway, gitignored).

### **Архитектурные и технические особенности**

1.  **Policy Gateway (Rust):**
    *   `agentwhite-core` загружает per-программный `policy.toml`: CIDR/домены scope, rate limit-лимиты, обязательные заголовки идентификации оператора, VPN-маркеры.
    *   Rate limit реализован через disk-based ledger — состояние лимитов сохраняется между перезапусками CLI.
    *   HTTP proxy-режим: httpx, nuclei, katana, ffuf маршрутизируются через gateway с инъекцией заголовков и ограничениями scope; passthrough-инструменты (gau, asnmap) получают переменные окружения без прямого проксирования.
    *   `agentwhite-cli` предоставляет подкоманды `preflight` (dry-run проверка policy и scope), `request` (единичный HTTP-запрос через gateway), `proxy` (локальный прокси), `tool <name>` (launcher с подстановкой proxy-флагов и переносом вывода в `.temporary/<program>/`).
    *   Аудит-лог пишется в `.enforcer/gateway/<program>/` — вне публичного слоя репозитория.

2.  **Research Log и управление состоянием кампании:**
    *   `programs/<name>/research-log.md` — хронологический журнал всей кампании: проверенные гипотезы, убитые векторы, активные priors, ключевые решения оператора и артефакты по фазам.
    *   Выдержки из лога включаются в промпт каждому новому субагенту, компенсируя нулевую стартовую память.
    *   Насыщенность лога служит сигналом к проектированию нового `TODO.md`; следующий TODO опирается на задокументированное состояние, а не на память оператора.
    *   `research-graph` (`graph.yaml` + Mermaid-представления) — разрабатываемый машинно-читаемый дополнительный артефакт для структурированного представления активов, векторов и находок.

3.  **Операторский контракт для ИИ-команды:**
    *   `AGENTS.md` — формальный контракт для агентов: миссия репозитория, явные запреты, карта инструментов и CI-контракт.
    *   `Techlead.mdc` — процесс главного агента: препродакшен → кластеры задач → вызов субагентов → Verifier → Verifier Pro. При блокерах или новых решениях Techlead эскалирует к оператору.
    *   `SubagentRules.md` — контракт субагентов: нулевая память при старте, приоритет scope, структура handoff-отчёта, классификация исходов (`finding`, `no finding`, `needs-human-repro`).
    *   Правила основаны на публичном фреймворке [AgentCompass](https://github.com/Artemonim/AgentCompass).
    *   Ростер субагентов: **OSINT Scout**, **OSINT Pro**, **Scope Sentinel**, **Surface Mapper**, **Vector Ranker**, **Field Operator**, **Traffic Triage** — каждый с конкретной ролью и местом в пайплайне.
    *   Рабочий цикл: Архитектор проектирует `TODO.md` → Techlead распределяет по кластерам и вызывает субагентов → каждый шаг пишется в `research-log` → при насыщении TODO проектируется следующий.

4.  **Мультиязычный монорепозиторий:**
    *   **Rust** — policy enforcement, core CLI, системные примитивы; выбор обусловлен необходимостью надёжного сетевого слоя без накладных расходов рантайма.
    *   **TypeScript** — типизированные веб-интеграции и tool-контракты; строгий ESM, `verbatimModuleSyntax`, ES2024.
    *   **Python** — аналитика и вспомогательная автоматизация с быстрой экосистемной интеграцией.
    *   **PowerShell** — CI-оркестратор и операторский слой в Windows-среде.
    *   **Go** — точечные утилиты там, где нужна компактная статически слинкованная поставка.

5.  **Recon Smoke CI и локальный оркестратор:**
    *   `run.ps1` — единая точка входа с флагами `-Fast` (только smoke), `-SkipTests`, `-SkipReconSmoke`.
    *   `Invoke-ReconToolsSmoke.ps1` поднимает локальный HTTP-сервер (`sandboxes/recon-smoke/www`) и прогоняет каждый инструмент по контрольному сценарию: httpx, katana, ffuf, nuclei (включая валидацию шаблона из субмодуля), subfinder/gau/asnmap, опционально naabu и Docker/BBOT.
    *   Языковые тесты объединяются в один прогон: `cargo test --workspace`, `pnpm test`, pytest.
    *   Итоговый JSON-отчёт (`.ci_cache/report.json`) фиксирует статусы стадий и длительности для повторного анализа.

## **Демонстрируемые навыки**

*   Проектирование сетевого enforcement-слоя на `Rust`: proxy, policy engine, disk-based rate limiter, аудит
*   Мультиязычный монорепозиторий: `Rust` + `TypeScript` + `Python` + `PowerShell` + `Go`
*   Проектирование многоуровневой ИИ-команды: Techlead, специализированные субагенты, handoff-контракт, ростер ролей
*   Накопление исследовательской памяти через research-log и per-program TODO-driven процесс
*   Policy-aware автоматизация для авторизованного bug bounty: scope, rate limit, аудит, VPN-маркеры
*   Двухрежимный доступ к поверхности: Cursor Browser MCP (пассивное наблюдение) + agentwhite gateway (контролируемые запросы)
*   Кураторский каталог внешних security-инструментов с smoke-верификацией через локальный сервер
*   Дисциплина артефактов и гигиена данных при работе с чувствительным контекстом
*   PowerShell CI/CD-оркестрация с машинно-читаемой отчётностью
